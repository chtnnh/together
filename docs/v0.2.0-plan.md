# Together v0.2.0 — Implementation Plan

**Status:** Approved scope · **Base:** v0.1.0 (main)  
**Audience:** Cloud agents / contributors — work through phases in order unless a task has no blockers.

---

## Goals

Ship a polished group-listening release focused on:

1. **Daily-use UX** — now-playing bar, keyboard shortcuts, reactions, mentions, connection feedback  
2. **Mobile** — touch drag-and-drop queue reorder (not overflow menus)  
3. **Playlists loop** — save queue / load playlist (auth)  
4. **Imports** — Spotify + SoundCloud E2E (Apple Music deferred)  
5. **Sharing & discovery** — OG cards, share sheet, invite tokens, public room directory, recent rooms  
6. **PWA** — service worker + manifest shortcuts  
7. **Infra** — CI deploy, rate limiting, DO ↔ Postgres settings sync

## UX decisions (do not regress)

| Topic | Decision |
|-------|----------|
| Mobile queue reorder | **Touch drag-and-drop** — same interaction model as desktop. Do **not** use a ⋯ overflow menu with move up/down (too many taps). Improve touch DnD (larger hit target, `touch-action`, optional long-press to drag). |
| Resync room state | **Button** in room header (e.g. “Sync” / refresh icon), **not** pull-to-refresh. |
| High-contrast theme | Add as **optional preset** — never the default. |
| PWA shortcuts | **Yes** — include in `manifest.json`. |
| @mentions | Autocomplete suggestions when user types `@`. |
| Batch queue ops | Good to have — implement if time after core queue/playlist work. |

## Phase 1 — Infrastructure

### 1.1 CI auto-deploy

**Files:** `.github/workflows/` (new `deploy.yml` or extend `ci.yml`)

- On push to `main` (after CI green): deploy Cloudflare Worker (`services/realtime`, `wrangler deploy --env production`).
- Vercel deploys web via Git integration (document required secrets).
- Use GitHub environments + secrets: `CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`, etc.

**Acceptance:** merge to main triggers worker deploy; web auto-deploys on Vercel.

### 1.2 Rate limiting

**Files:** `apps/web/src/app/api/rooms/route.ts`, `apps/web/src/app/api/import/youtube/route.ts`, possibly middleware

- Rate limit room creation per IP (e.g. 10/hour).
- Rate limit YouTube import/search per IP.
- Return `429` with clear JSON error.

**Acceptance:** rapid abuse returns 429; normal use unaffected.

### 1.3 Durable Object ↔ Postgres settings sync

**Files:** `services/realtime/src/room-do.ts`, `apps/web/src/lib/rooms.ts`, optional cron/API route

- When host updates settings via WS, client already PATCHes `/api/rooms/[slug]/settings` if room has owner.
- Ensure DO settings sync to DB on host `settings:update` (HTTP callback from DO **or** rely on client PATCH + fix gaps for claimed rooms).
- On DO init, do **not** clobber live settings (already fixed in 0.1 — verify still true).

**Acceptance:** host changes settings → refresh → settings persist; DB matches DO for owned rooms.

## Phase 2 — Spotify + SoundCloud import (E2E)

### 2.1 Spotify

**Existing:** `apps/web/src/app/api/auth/spotify/`, `apps/web/src/app/api/import/spotify/`, `apps/web/src/app/import/spotify/`, `apps/web/src/lib/spotify.ts`

- Audit OAuth flow end-to-end in production-like env.
- Fix broken paths; add error toasts on failure.
- Room flow: “Import Spotify” → pick playlist → tracks land in requests/queue per role/settings.
- Document env vars in `.env.example`.

**Acceptance:** user with Spotify OAuth can import a playlist into a live room; tracks resolve to YouTube where possible.

### 2.2 SoundCloud

**New work:** mirror Spotify pattern

- OAuth or API-key approach (evaluate SoundCloud API; may need client credentials for public tracks).
- `apps/web/src/lib/soundcloud.ts`, `apps/web/src/app/api/import/soundcloud/`, auth routes if needed.
- Resolve SoundCloud tracks → YouTube via existing `@together/track-resolver` (title/artist search).
- UI: “Import SoundCloud” button next to Spotify in room sidebar (`room-client.tsx`).

**Acceptance:** paste SoundCloud playlist/profile URL or OAuth flow → tracks added to room queue/requests.

### 2.3 Apple Music

**Do not implement in 0.2.** Leave existing code; hide or disable “Import Apple Music” button until v0.3.0.

## Phase 4 — Now playing & input

### 4.1 Now-playing card

**Files:** new `apps/web/src/components/now-playing-bar.tsx`, integrate in `room-client.tsx`

- Pinned strip (especially visible in audio-only mode): thumbnail, title, seek, play/pause, skip.
- Works on mobile + desktop; does not duplicate entire playback controls block awkwardly.

### 4.2 Keyboard shortcuts

**Files:** `room-client.tsx` or `use-keyboard-shortcuts.ts`

- `Space` — play/pause (when controls allowed, not typing in input)
- `←` / `→` — seek ±5s
- `n` — skip / vote skip?
- `/` — focus add URL input
- `?` — show shortcuts help overlay

**Acceptance:** shortcuts work on desktop; ignored when focus in input/textarea.

### 4.3 Now-playing reactions

**Files:** `packages/shared/src/events.ts`, `services/realtime/src/room-do.ts`, UI component

- New event e.g. `reaction:send` with emoji enum (🔥 👏 💀 ❤️ 😂).
- Broadcast to room; show floating reactions on now-playing bar (auto-fade).
- Rate limit per user (e.g. 1 per 2s).

### 4.4 Crossfade / gapless

**Files:** `use-youtube-player.ts`, `room-do.ts` (`advancePlayback`)

- Short crossfade (~300–500ms) or gapless handoff between queue items where YouTube API allows.
- Note: true gapless may be limited by YouTube iframe — document limitations; best-effort.

### 4.5 Volume normalization

**Files:** `use-youtube-player.ts`

- Research YouTube player volume API; normalize perceived loudness between tracks if feasible (may be heuristic: start at consistent volume level after track load).
- Best-effort; document if not fully achievable via iframe API.

### 4.6 Background playback (mobile)

**Files:** `use-youtube-player.ts`, `room-client.tsx`

- Minimizing the browser or switching apps must **not** pause synced playback (common issue on mobile Safari/Chrome).
- Use Page Visibility API + resume logic when the tab becomes hidden/visible; keep player state aligned with room playback.
- Do not pause locally when `document.hidden` unless the room itself is paused.

**Acceptance:** on mobile, background the tab while a track is playing — audio continues (or resumes within ~1s of foregrounding without user tap).

### 4.7 Desktop audio-only hides video

**Files:** `room-client.tsx`

- When **Audio only mode** is enabled, hide the YouTube iframe on **desktop** as well as mobile (today video still shows on `md+`).
- Show the existing audio-only now-playing placeholder on all breakpoints.

**Acceptance:** toggle audio-only on desktop — no video iframe visible; title/controls remain usable.

## Phase 5 — Queue & requests

### 5.1 Democratic promote UI

**Files:** `packages/ui` request list, `room-client.tsx`, `room-do.ts` (already has `vote:promote`)

- Vote button on requests when `settings.democraticPromote` is on.
- Show vote count + threshold (mirror skip vote bar pattern).

### 5.2 Touch drag-and-drop queue reorder

**Files:** `packages/ui/src/components/queue-list.tsx`

- Improve mobile touch DnD: `touch-action: none` on drag handle, pointer events, visual lift on drag.
- Consider `@dnd-kit/core` if native HTML5 DnD is unreliable on iOS — **prefer one dependency over ⋯ menus**.
- Host/co-host only (same as desktop).

**Acceptance:** single continuous drag gesture moves a track on iOS Safari + Android Chrome.

### 5.3 Batch queue ops (if time)

- Multi-select on requests/history → promote all / re-add all.
- Lower priority than 5.1 and 5.2.

## Phase 6 — Playlists (auth)

**Files:** `apps/web/src/app/api/playlists/route.ts`, room UI

- Host/co-host button “Save queue” → name dialog → POST playlist with current queue items.

### 6.2 Load playlist into room

**Files:** `apps/web/src/app/playlists/`, room sidebar or settings

- Adds all resolved tracks to queue/requests per settings.

### 6.3 Sync user prefs to account

**Files:** `use-user-preferences.ts`, new API route, DB schema if needed

- Store theme, audioOnly, quality in user profile (Supabase) when signed in.
- Load on sign-in; fallback to localStorage for anonymous.

### 6.4 Room ownership transfer

**Files:** `room-do.ts`, `apps/web/src/app/api/rooms/[slug]/route.ts`, participants panel

- Host can transfer ownership to signed-in co-host or guest.
